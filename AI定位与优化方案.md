# CleanScope AI 定位与优化方案（第三轮 · 给 AI 一个对的位置）

> 承接 [产品问题与改进方案.md](产品问题与改进方案.md)（第一轮）与 [清理逻辑重构方案.md](清理逻辑重构方案.md)（第二轮，含 S-C 让 AI 调查未知项、已落地）。
> 本轮不再"给 AI 加戏"，而是回答一个更根本的问题：**在这个工具里，AI 到底适合做什么、不该做什么。**
>
> **一句话结论**：AI 应当是 **「识别官 + 翻译官 + 全局参谋」**——把"未知变已知、难懂变好懂、零散变全局"；
> 它**永不当裁判**（不碰"能不能删"），也**不该做依赖发掘**（那是可验证硬事实，归确定性引擎）。

---

## 一、现状诊断：AI 现在为什么"形同摆设"

三层原因叠加，前两层是工程问题、第三层是定位问题：

1. **架构上被刻意架空（这是对的安全红线）**
   裁决链 `扫描→证据→规则→归因→风险→决策`（[ScanAndAnalyzeUseCase](src/CleanScope.Application/ScanAndAnalyzeUseCase.cs)）是 **100% 确定性本地逻辑**。
   [AiOutputValidator](src/CleanScope.Ai/Validation/AiOutputValidator.cs) 还会夹逼 AI：不能调低风险、引擎判 E 强制还原 E、不能开删除绿灯。
   → AI 唯一能产出的是**解释性文字**，改变不了任何结论。

2. **默认是关的 → 对没配密钥的用户贡献 = 0**
   [CompositionRoot](src/CleanScope.App.Wpf/Composition/CompositionRoot.cs) 里 `aiEnabled = aiOptions.IsUsable`，无 API key 即 `false`，全程本地。
   多数用户不会配 `appsettings.ai.local.json` → AI 在他们那儿**完全不存在**。这是"摆设感"最直接的来源。

3. **只会"逐项解释"，没做它真正擅长的事**
   现在 AI 是 per-item（详情页单项 / E 项逐个，[AiAnnotator](src/CleanScope.Application/AiAnnotator.cs)）。但逐项判断确定性规则也能做；
   AI 的独特价值在**跨项、靠世界知识、容错**的任务上——而这些一个都没做。

---

## 二、判据：什么该给 AI，什么不该

用一把尺子量：**AI 只该出现在答案 (1) 难以用规则穷举、(2) 是建议性/容错、(3) 靠世界知识 的地方；
凡是"可验证硬事实 + 安全关键"，一律留给确定性引擎并 fail-safe。**

| 维度 | 适合确定性引擎 | 适合 AI |
|---|---|---|
| 答案来源 | 注册表/签名/句柄/包清单/文件系统（可查、可复现） | 世界知识、常识、自然语言 |
| 错了的代价 | 高（删错=毁数据） | 低（只是建议，不触发删除） |
| 可枚举性 | 有限集合，规则能覆盖头部 | 长尾，规则永远有缺口 |
| 形态 | 精确事实（大小/路径/占用/定级） | 模糊解释、跨项综合、归类 |

### 为什么"依赖发掘"不该给 AI（本轮触发点）

依赖关系恰好三条全踩 LLM 短板：

- **是可验证硬事实**：真正的依赖证据来自注册表卸载项、已安装应用库、包清单、文件句柄占用、硬链接、数字签名——
  [WindowsAccess](src/CleanScope.Infrastructure/Windows/WindowsAccess.cs) / [RestartManager](src/CleanScope.Infrastructure/Windows/RestartManager.cs) 已能确定性取到。
- **判错=出事**：让 LLM 猜"X 是否被 Y 依赖"是不可验证的幻觉，猜错少删=留垃圾、猜错多删=毁数据。
- **正是 fail-safe 不敢信 AI 的地方**：这与"校验器不许 AI 调低风险"同源。

> 注意区分两种"依赖"：① **硬依赖事实**（句柄/注册表/硬链接）——确定性，AI 无用且有害；
> ② **可重建性常识**（浏览器缓存可重建、node_modules 可重装）——规则包已覆盖头部，AI 只在**长尾未知**时补充，且仅作建议。
> 无论哪种，"依赖发掘"都不该是 AI 的头号任务。**你的直觉是对的。**

## 三、AI 的新定位与适配排序

| 适配 | 任务 | 为什么是 AI 的活 | 现状 |
|---|---|---|---|
| **★★★** | **识别官**：长尾"这是什么、归哪个软件" | 归因靠 [AttributionEngine](src/CleanScope.Core/Attribution/AttributionEngine.cs) 正则+厂商字典，**永远有缺口**；"目录名→软件"是海量长尾世界知识，字典覆盖不了、AI 训练数据覆盖。容错、不碰删除 | 仅 E 项零散调查 |
| **★★★** | **全局参谋**：整盘跨项清理叙事 | "装了三套 Python、缓存重复 4GB""这些是 IDE 自动产物，通常可重建"——跨项常识综合，规则引擎天生做不了，只给建议 | 无 |
| **★★☆** | **翻译官**：把判定翻成人话 / 交互问答 | 基于工具已采集证据（grounding）用自然语言解释"删了会怎样"，事实由本地提供、AI 只做表达 → 幻觉低 | 详情页单项解释（已有） |
| **✗ 不做** | 依赖发掘 / 风险定级 / 大小·占用·签名 | 可验证硬事实 + 安全关键 → 确定性 + fail-safe | 已正确归确定性 |

## 四、优化方案（逐项：做什么 → 锚点 → 安全）

### S-G　归因兜底召回（识别官）— ★★★

- **做什么**：确定性归因跑完后，对**无主 / 弱主**且体量不小的目录，批量请 AI 给"归属软件 + 用途"推测；
  以 **`Source="AI推测"`、置信度上限封顶（如 ≤0.5）** 的形式补进 [AttributionCandidate](src/CleanScope.Domain/Models/Models.cs)。
- **效果**：直接削掉「按软件」视图里的"未归类 / 未知来源"大桶；用户终于"认得出"那些杂目录。
- **安全**：AI 归属**只用于展示与分组**，封顶置信度**够不到** RiskEngine 里影响定级的阈值，**永不抬 `canDelete`、永不降风险**；UI 明确标注"AI 推测归属"。

### S-H　整盘清理参谋（全局参谋）— ★★★

- **做什么**：扫描后对**聚合结果**（[SoftwareAggregator](src/CleanScope.Reporting/SoftwareAggregator.cs) + 可清理类别 + TopN，全部脱敏）跑**一次**整盘 AI，
  产出跨项发现：冗余工具链、重复缓存、"通常安全可清理"的归纳、优先级建议。
- **落地**：写进报告新「🧭 AI 清理参谋」节 + 首页一张卡片。**一次调用、成本可控**。
- **安全**：纯建议文字；任何删除仍走详情页"逐项 + 闸门 + 两步确认 + 审计"。AI 不产出可执行删除指令。

### S-I　AI 知识库沉淀（让 AI 越用越值、且离线也在）— ★★☆

- **做什么**：把 AI 的"识别"结果按**脱敏路径模式**持久化进本地 SQLite（复用 [AiAnnotator](src/CleanScope.Application/AiAnnotator.cs) 的缓存键思路），跨扫描复用；
  对**反复高置信命中**的模式，生成"规则包候选"供人工 review 后固化为确定性规则。
- **效果**：一次云端调用 → 变成可复用的本地归因增强；首扫后即便离线，AI 的识别成果仍在。token 成本随使用下降。
- **安全**：沉淀的是**归因/解释**，非删除决策；固化为规则仍走规则包既有校验。

### S-J　离线兜底（无密钥也别全空）— ★★☆

- **做什么**：无 key 时，用**轻量本地映射 / 启发**覆盖最常见长尾目录名（与字典互补），让"AI 形状的价值"不再是全有或全无。
- **安全**：本地启发，等价于扩充确定性归因，零出云。

### S-K　交互问答（翻译官 · grounded）— ★☆

- **做什么**：详情页/列表选中项问"这能删吗/为啥这么大"，AI 基于**本次扫描已采集的证据** RAG 式作答。
- **安全**：答案由本地事实 grounding，AI 只解释；不改判、不删除。UI 工作量较大，置后。

### S-L　可观测性与质量抽检 — ★☆

- **做什么**：记录 AI 调用数 / 缓存命中率 / token / 失败率；建一个**手动触发、默认跳过**的归因准确率抽检集。
- **价值**：让"AI 帮了多少、值不值"可度量，而非凭感觉。

## 五、落地路线图与验收

| 步 | 任务 | 价值 | 风险 | 依赖 |
|---|---|---|---|---|
| **1** | S-G 归因兜底召回 | 高（消灭未归类大桶） | 低 | 无 |
| **2** | S-H 整盘清理参谋 | 高（AI 终于"不可替代"） | 低 | 无 |
| **3** | S-I 知识库沉淀 | 中高（越用越值、离线在） | 中 | S-G |
| **4** | S-J 离线兜底 | 中（默认非零贡献） | 低 | 无 |
| **5** | S-K 交互问答 | 中 | 中 | S-I |
| **6** | S-L 可观测性/抽检 | 中 | 低 | 无 |

**验收（关键）**：
- 「按软件」视图中"未归类 / 未知来源"占比显著下降（S-G）。
- 报告/首页出现至少 1 条**跨项**建议（冗余工具链/重复缓存类），是规则引擎产不出的（S-H）。
- 第二次扫描相同机器时 AI 调用数明显下降（命中知识库，S-I）；无 key 时仍有基础识别（S-J）。
- 全程：AI 归属/建议**永不**抬升 `canDelete`、**永不**降风险、**永不**产出可执行删除指令——有用例守。

> 建议先做 **步骤 1 + 2**：它们一个解决"认不出"，一个让用户**直观感到"这确实得靠 AI"**，且都在安全边界内。

## 六、安全宪法影响评估（逐条确认不破红线）

| 红线 | 本轮是否触碰 | 说明 |
|---|---|---|
| ① AI 不直接删除重要文件 | 否 | AI 仅识别/解释/参谋；删除仍逐项 + 闸门 + 两步确认 + 审计 + 仅回收站可恢复 |
| ② 优先解释来源/依赖/风险 | 强化 | 识别官/参谋正是"优先解释" |
| ③ 用户做最终删除决定 | 保持 | 不变 |
| 风险定级权威性 | 保持 | AI 归属封顶置信、永不降风险、永不开删除绿灯（[AiOutputValidator](src/CleanScope.Ai/Validation/AiOutputValidator.cs) 既有夹逼） |
| 依赖/占用等硬事实 | 保持 | 仍由确定性引擎取证，AI 不参与 |
| 无永久删除 API | 保持 | 与 AI 无关；删除仅回收站可恢复（[WindowsRecycleBin](src/CleanScope.Infrastructure/Windows/WindowsRecycleBin.cs)） |
| 脱敏出云 | 保持 | 整盘参谋/归因仍走脱敏网关；知识库存脱敏模式 |
| CI 门禁 | 扩充 | 新增"AI 归属不抬删除/不降风险""AI 不产可执行删除指令"等用例 |

---

## 七、一句话总纲

> **确定性引擎当裁判（定级、依赖、能不能删）；AI 当识别官 + 翻译官 + 全局参谋（这是什么、归谁、整盘怎么收拾）。**
> 各司其职、职责不重叠——这才是 AI 在 CleanScope 里"对的位置"。
