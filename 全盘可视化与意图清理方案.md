# CleanScope 全盘可视化与意图驱动清理方案（第四轮 · 让用户真正"看清并清掉"）

> 基于一次真机使用后的反馈。前三轮把"统计可信 / 归因清楚 / 删除可恢复 / AI 各司其职"做扎实了，
> 但**产品形态**仍是"扫描→给一张 Top-N 风险清单"，而用户要的是**一个能自由浏览、下钻、按自己意图清理的资源管理器**。
> 本轮是形态级重构。

---

## 一、用后真实痛点（逐条 → 根因）

| # | 痛点（用户原话） | 根因 |
|---|---|---|
| **1** | 概览 Top10 没信息价值，全是 `C:\`、`Users`、`AppData`、`Windows` 这种很重要的大文件夹 | Top10 按**绝对大小**排，最大的永远是容器/系统 —— **"最大"≠"最该处理"** |
| **2** | 文件清单只显示 200 个；别的软件能把整个 C 盘当**目录树**展示，可展开/折叠，旁边显示大小、按大小排序 | 扫描只保留 **Top-N 节点**（[ScanEngine](src/CleanScope.Core/Scanning/ScanEngine.cs) 返回 TopN），前端没有**完整目录树** |
| **3** | 可清理的太少 | 风险定级**对所有人保守**（应用数据一律 C）；真正的可清理（app 内部缓存）**没下钻识别** |
| **4** | `…\Tencent\xwechat` 里很多子文件夹，没检测这些子文件夹能不能删 | 决策只到**被选中的目录**，没递归判断**子目录**里哪些是缓存可删、哪些是数据要留 |
| **5** | 不想用某软件时，C 级风险对他其实是 A 级（可删） | 工具只有"**安全维度**"，缺"**用户意图维度**"——没有"我不再需要 X"这个输入 |
| **6** | 不清楚整个 C 盘各文件占多大、各干什么 | = 痛点 1+2 的合并：缺**全貌 + 逐项用途**的统一视图 |

**一句话**：现在是"风险审计器"，用户要的是"**磁盘资源管理器 + 我说了算的清理**"。

---

## 二、重构目标

把主界面从「Top-N 风险清单」升级为「**全盘目录树**」，并让删除由**安全等级 × 用户意图**共同决定：

> **工具负责**："这是什么、占多大、删了会怎样、是否安全"（确定性 + AI 补长尾）。
> **用户负责**："我还要不要它"。两者结合 → 高效、自由、清晰地清理。

---

## 三、五大支柱（逐条：做什么 / 为什么 / AI 用不用）

### P1　全盘目录树（核心，解痛点 1/2/6）

- **做什么**：扫描保留**完整目录树**（每个目录节点：聚合大小、子目录数、子节点），前端用**可展开/折叠的树**（TreeView，惰性展开、虚拟化）：
  - 每个节点旁显**大小**、占父目录**百分比**、横向**大小条**；同级**按大小降序**。
  - 节点带本轮已有的**来源/归属 + 用途**（[ContainerPurpose](src/CleanScope.Core/Attribution/ContainerPurpose.cs)/[SystemOrigin](src/CleanScope.Core/Attribution/SystemOrigin.cs)/归因）。
  - 展开某目录时**按需列出其文件**（文件不必全驻内存）。
- **为什么**：这正是 WizTree/TreeSize/SpaceSniffer 的核心体验——用户能**从 `C:\` 一路点进去**，看清"空间到底去哪了"，自己找到要清的东西。
- **AI**：**不用**（纯计算：大小、排序、树）。

### P2　子目录可清理下钻（解痛点 3/4）

- **做什么**：把"可清理识别"**递归到任意深度的子目录**，区分**可重建缓存** vs **真实数据**：
  - 扩充确定性规则：在**任意 app 目录下**命中通用缓存子目录名 →（B 可清理）：
    `Cache / Code Cache / GPUCache / blob_storage / Service Worker / CachedData / CacheStorage / logs / crashpad / tmp / Temp / GrShaderCache / DawnCache …`（你截图里 `WeChat\XPlugin\Plugins`、`Code\Cache`、JetBrains/浏览器缓存都属此类）。
  - 账号/消息/配置/数据库 → 保留（C，明确标"删了丢登录态/聊天记录"）。
- **为什么**：这才是"可清理太少"的真出口——一个 5 GB 的 Tencent 里，往往有**好几 GB 是可重建缓存**，现在被整体标 C 埋没了。
- **AI**：**用（高价值）**——规则覆盖不到的**陌生子目录**，AI 判"**这是缓存还是真数据、删了会不会丢东西**"（可重建性判断），按需触发、明确标 AI 推测。这是 AI 真正不可替代的地方。

### P3　意图驱动清理：「我不再使用这个软件」（解痛点 5）

- **做什么**：把"安全等级"和"用户意图"**解耦**。「按软件」视图每个软件加动作：
  - **「我不再使用 → 移除其数据」**：列出该软件**所有目录**（数据/缓存/安装），强确认后**整体移入回收站**（可还原）；安装目录跳**卸载程序**。
  - 工具仍只标安全（系统关键/占用**永不**让步），但用户**显式表达意图**后，C 级应用数据变为"可按你的意图清理"。
- **为什么**：直接回应"不想用某软件时 C 对他就是 A"。让用户用**自己的知识**（我不用微信了）补上工具无法替代的判断。
- **AI**：**不用**于"要不要删"（那是用户的）；但用 P2 帮用户看清"删这个软件会丢什么、哪些只是缓存"。

### P4　把概览换成"可操作"而非"最大"（解痛点 1）

- **做什么**：概览不再罗列最大的容器，而是给**可下手**的东西：
  - ①「可清理估算 + 一键去清理类别」；②「按软件占用 Top（各能清多少）」；③「**最大的非容器/非系统叶子目录**」（真正占地方又能下钻的）；④AI 整盘参谋（[已有 S-H](src/CleanScope.Ai/Advice/CleanupAdvisor.cs)）。
  - "Top-N by size"挪进**树视图**里——在树里看最大才有意义（能展开）。
- **AI**：参谋已有，保留（有价值）。

### P5　删除策略扩展（解痛点 3/5，**安全敏感，需你拍板**）

- **现状**：安全闸门只放行 **A/B** 进回收站（[SafetyGuard](src/CleanScope.Safety/SafetyGuard.cs)）。
- **提议**：把"**C 级应用数据**"（非系统关键 / 非容器 / 未占用）纳入**可回收站删除**，但要求**逐项/逐软件显式确认**。理由：C 是"谨慎"不是"危险"，且**仅进回收站可还原**；用户意图明确时不该拦着。
- **永不让步**：D / 系统关键黑名单 / 容器整体 / 占用中 → 仍**硬拒**；仍**无永久删除**；删除前**审计**；批量需逐条/分组二次确认。
- **AI**：**不参与**删除判定（红线）。

## 四、AI 定位总表（对齐"有价值才用"）

| 用 AI（不可替代） | 不用 AI（确定性/用户主导） |
|---|---|
| P2 子目录"缓存 vs 数据 / 可重建性"长尾判断 | 目录树、大小、排序、百分比（纯计算） |
| 未知项调查（[S-C](src/CleanScope.Application/ScanAndAnalyzeUseCase.cs)，已有） | 风险定级 / 安全门禁（确定性 + fail-safe） |
| 归因兜底（[S-G](src/CleanScope.Core/Attribution/AttributionEngine.cs)，已有） | 系统/容器来源与用途（已是确定性表） |
| 整盘清理参谋（[S-H](src/CleanScope.Ai/Advice/CleanupAdvisor.cs)，已有） | "要不要删"（用户意图） |

> **顺手做的减法**：审查现有 AI 触发点，确保**只在"规则无法判定"时**才调 AI（详情页对"明显是缓存/明显是系统"的项不该浪费调用）——避免"为用而用"。

## 五、架构改动（落地锚点）

1. **扫描产出全目录树**：[ScanEngine](src/CleanScope.Core/Scanning/ScanEngine.cs) 从"返回 Top-N 文件"改为"构建**目录树**（`DirNode{ Path, Size, FileCount, Children }`）+ 仍给 Top-N 作快捷入口"。目录数量远小于文件数（万级），可全驻内存；文件级展开时按需读。
2. **决策链下钻**：对**目录树节点**逐个跑 证据/规则/归因/风险（或惰性按展开跑）；P2 的通用缓存规则在此生效。
3. **存储可选**：大盘可把节点持久化到 SQLite（[FileNodeRepository](src/CleanScope.Infrastructure/Repositories/FileNodeRepository.cs) 已存在），UI 从 DB 惰性查询，控内存。
4. **新「资源管理器」树视图**（WPF）：TreeView + 虚拟化 + 惰性展开 + 大小条；复用现有 [TreeBuilder](src/CleanScope.App.Wpf/ViewModels)/空间地图的下钻逻辑，但喂**全树**而非 Top-200。
5. **性能**：现状 115 万项约 2 分钟可接受；树在扫描时顺带聚合，不额外加倍。

## 六、安全宪法影响

| 红线 | P1–P4 | P5 |
|---|---|---|
| 无永久删除（仅回收站可恢复） | 不触碰 | **保持**（仍仅回收站） |
| 系统关键/黑名单恒拒 | 不触碰 | **保持**（D/系统/容器/占用硬拒） |
| 用户做最终删除决定 | 强化（树+意图） | **强化**（逐项/逐软件显式确认 + 审计） |
| AI 不驱动删除 | 保持 | 保持 |

> 仅 **P5** 动"可删范围"（C 应用数据），且只到回收站、需显式确认——需你选档（见下）。

## 七、落地路线图

| 阶段 | 任务 | 解决 | AI | 风险 |
|---|---|---|---|---|
| **1** | 扫描产出**全目录树** + （可选）持久化 | 2/6 基础 | 否 | 中（核心改动） |

## 八、验收

- 能从 `C:\` 一路展开到任意深度，每个节点有**大小 + 排序 + 来源 + 用途**；不再有 200 项上限。
- 概览第一屏给的是"**能清什么、能清多少**"，而非最大的系统文件夹。
- 进到 `…\Tencent\xwechat` 能看到**内部哪些子目录是可清理缓存**（而非整体一个 C）。
- 「按软件」可对"不再使用的软件"一键（强确认 + 回收站）清其数据。
- 全程红线不破；若做 P5，C 应用数据删除仅回收站、需逐项确认、随审计与用例。

## 九、需要你拍板

**P5 删除范围三选一**（其余 P1–P4 我都会做）：

- **A. 保持现状**：仅 A/B 可进回收站；C 只读引导（最保守）。
- **B. 扩到 C 应用数据**：C 级**应用数据**逐项确认后可进回收站（系统/容器/占用仍硬拒）。
- **C. B + 按软件整体移除**：再加「我不用这软件了 → 整体移入回收站 + 跳卸载」。
