# CleanScope 清理逻辑重构方案（第二轮 · 针对现有功能缺陷）

> 承接 [产品问题与改进方案.md](产品问题与改进方案.md)（第一轮 6 步已落地）。本文基于 2026-06-17 两次真机 C 盘扫描，
> 针对**当前仍存在的功能缺陷**给出逐条优化方案与落地路线。
> 取证报告：[report/CleanScope-报告-20260617-150540.md](report/CleanScope-报告-20260617-150540.md)（Admin，167.75 GB / 115.7 万项）。
>
> **核心结论**：第一轮解决了"统计可信 + 挖出可清理空间 + 性能"，但**用户心智与可操作性**仍差一截 ——
> 「无法判断太多」「不知道每个目录归谁」「能删的还得自己去找去删」。本轮聚焦把工具从"看得懂"做到"用得动"。

## 一、缺陷清单（逐条：证据 → 根因 → 影响）

### D1　AI 在整条检测中形同摆设

- **证据**：报告「分级明细」的**说明**列全部来自规则/风险引擎（`目录名表明为可重建缓存…`、`有官方清理方式; 规则 nuget-packages`），**无一条出自 AI**。
- **根因**：桌面端扫描用 `AiMode.OnDemand`（第一轮性能优化），扫描全程不调用 AI；AI 仅在**打开单个文件详情页**时临时生成解释，且**不写回**会话/报告/列表。
- **影响**：整条流水线 100% 确定性本地逻辑，AI 对"是什么/能不能删/占多大/归谁"零贡献，沦为详情页的临时讲解员。安全上没问题（AI 本就只能建议），但**产品价值上 AI 当前近乎摆设**。

### D2　"无法判断"泛滥之一：顶层容器被当成删除对象

- **证据**：E 级高风险列表里赫然是 `C:\`、`C:\Users`、`C:\Users\28170`、`C:\Users\28170\AppData`、`AppData\Roaming`、`AppData\Local`、`C:\Program Files`、`C:\Program Files (x86)`、`C:\ProgramData` —— 全标「无法判断, 不建议删除」。
- **根因**：风险引擎的路径启发式（`LocalAppDataRx` 等）要求路径在容器名**之后还有一段子目录**才命中；容器目录本身没有子段 → 落入兜底 E。
- **影响**：这些**根本不是删除候选**（是"装东西的柜子"），却占据 TopN 与 E 榜首，制造大量噪声，让用户误以为"一堆东西无法判断"。

### D3　"无法判断"泛滥之二：有主目录因归因阈值掉进 E

- **证据**：`C:\Users\28170\.cargo`、`.nuget`、`.rustup`、`.rustup\toolchains`、`Miniconda3`、`.lingma`、`vscode-remote-wsl` 全是 E；但其**子目录** `.cargo\registry`、`.nuget\packages` 却是 B 且归属正确（Rust/Cargo、NuGet）。
- **根因**：[RiskEngine.cs](src/CleanScope.Core/Risk/RiskEngine.cs) 仅在 `attributions.Any(a => a.Confidence >= 0.8)` 时用归因避免 E；而路径模式归因置信度只有 **0.5**（[AttributionEngine.cs](src/CleanScope.Core/Attribution/AttributionEngine.cs)），于是"明明认识它归谁"却仍判 E。
- **影响**：**"有主"却"无法判断"**自相矛盾。用户最想要的"这目录归哪个软件"信息被 E 的笼统措辞淹没。

### D4　归因未前置：风险表不优先回答"归谁"

- **证据**：报告主结构是「按风险等级」分组；归属只是明细表的一列，且 E 项归属常为 `-`。
- **根因**：决策汇总 [DecisionService.cs](src/CleanScope.Core/Decisions/DecisionService.cs) 以 `RiskLevel` 为第一排序键；UI 也以风险为主轴。
- **影响**：用户的第一诉求是"**这是谁的、我认不认识**"，而非抽象的 A–E。归因没当成一等公民。

### D5　A/B 可清理项不可操作：还得自己找路径、自己删

- **证据**：`Miniconda3\pkgs` 给了 `用 conda clean --all`，但用户得自己开终端、自己敲；文件夹型缓存只给"建议用官方方式清理"，得自己去资源管理器翻。
- **根因**：详情页只有「打开文件夹/复制路径/加忽略」三个弱动作；命令是纯文本，不能一键运行；列表/类别视图无任何动作入口。
- **影响**：**"知道能删"到"真的删掉"之间断了一截**。这正是你说的"AB 等级还得手动去找路径"。

### D6　风险等级 A–E 对普通用户不直观

- **证据**：A–E 是内部模型；用户问的是"能删吗？怎么删？"。
- **根因**：UI 直接暴露内部风险枚举。
- **影响**：认知负担高；C(127 项)这种"谨慎"大桶里混了"应用数据"和"安装目录"两种很不同的东西，指引含糊（统一文案"建议先备份或确认用途"）。

### D7　命令型 vs 文件夹型清理未区分

- **证据**：`conda clean --all`（命令型）与 `Code\CachedExtensionVSIXs`（文件夹型，无命令）都归到 B，给一样的"走官方方式"。
- **根因**：规则里 `recommended_action` 是自由文本，未结构化区分"可执行命令"与"手动/定位"。
- **影响**：无法对命令型做"一键运行"，对文件夹型做"一键定位/回收站"。

## 二、重构核心：四桶心智模型 + 归因前置

把每一项从抽象 A–E 映射到**四个可操作的桶**（A–E 仍作内部权威，但 UI 以桶呈现）：

| 桶 | 内部对应 | 含义 | 该给的动作 |
|---|---|---|---|
| **🗂 容器（仅浏览）** | 新增判定 | 顶层目录，装着多个软件的数据 | 「展开/下钻」，**不进"待清理"，不算风险** |
| **✅ 可清理** | A / B | 缓存 / 包 / 临时，归属明确、可重建 | **命令型→一键运行/复制官方命令；文件夹型→一键定位 / 移入回收站** |
| **⚠ 谨慎** | C | 应用数据 / 安装目录，归属明确 | 「打开软件设置 / 卸载程序」+ 明确归属 |
| **🛑 勿动** | D / E（真未知） | 系统关键 / 真正来源不明 | 只展示原因，**无任何删除入口** |

三条主线：
1. **先回答"归谁"**：归因前置，几乎每个非系统目录都该有主（D3/D4）。
2. **再回答"能不能删"**：四桶比 A–E 直观（D6）。
3. **最后把"怎么删"做成按钮**：命令型/文件夹型分别给动作（D5/D7）。

## 三、逐项解决方案

### S-A　归因前置 + "有主即非 E"（解 D3/D4）

- **风险引擎**：把"E 兜底前"的归因门槛从 `≥0.8` 降为"**存在任意归因候选**"即 → **C**，因子写明`归属 <App>，暂无清理方式，谨慎`。`.cargo`/`Miniconda3`/`.lingma` 等从 E 升 C 且显示归属。
- **决策/UI**：归因（归属应用）提到一等位置；提供「**按软件聚合**」视图（见 S-F）。
- **归因增强**：扩充厂商目录字典（飞书/腾讯/Notion/JetBrains 已有，补充 WeChat/Edge/Adobe/NVIDIA/Lenovo/Docker/WSL 等）；`.lingma`、`vscode-remote-wsl` 等加入字典。

### S-B　容器识别与降噪（解 D2）

- **风险引擎**：识别"顶层容器目录"集合 —— 盘符根、`C:\Users`、`C:\Users\<user>`、`AppData`/`AppData\Local`/`AppData\LocalLow`/`AppData\Roaming`、`Program Files`/`Program Files (x86)`、`ProgramData`、`Windows` —— 标为新桶 **容器（仅浏览）**，**不计入风险统计、不进"高风险提醒"与"可清理"**。
- **呈现**：容器在**空间地图**里天然就是下钻入口；在列表/报告里要么折叠、要么单列「容器」分区，文案`内含多个程序的数据，请展开按子目录判断，不要整体处理`。
- **实现选项**：① 在 `RiskAssessment` 加 `IsContainer` 标记（最清晰）；② 或决策层按路径集合过滤/分区。倾向 ①。

### S-C　给 AI 一个真实定位（解 D1）

AI 不进裁决（红线），但可以从"摆设"变"**未知项的按需调查员**"：

- **详情页（已有，强化）**：打开 E/C 项时，AI 给"这可能是什么、归哪个软件、通常能不能清"的**推测**（明确标注"AI 推测，仅供参考，不改变风险"）。
- **可选·针对 E 项的批量增强**：扫描后**只对少量 E 项**（如 TopN 里的 E）并发+缓存跑一次 AI（复用第一轮 `AiAnnotator`），把推测写进报告的"调查建议"列。代价可控（E 项通常几十个），让 AI 真正参与"消化无法判断"。
- **定位明确**：AI 负责"解释与调查未知"，规则/风险负责"权威判定"，两者职责不重叠。

### S-D　动作闭环：能删的怎么快速删（解 D5/D7）

把规则的 `recommended_action` **结构化**，新增动作模型（领域层）：

```
ActionHint {
  Kind:  RunCommand | OpenInExplorer | OpenSettings | MoveToRecycleBin | None
  Command?: string        // 命令型: 如 "conda clean --all"
  SettingsUri?: string    // 设置型: ms-settings: / 应用内路径说明
  TargetPath: string
}
```

- **命令型缓存** → 详情/列表/类别视图都给「**复制命令**」+「**在终端运行**」（`cmd /c <command>`，由厂商工具安全清理；我们不删文件）。
- **文件夹型缓存** → 「**在资源管理器中定位**」（打开并选中）+（视删除决策）「**移入回收站**」。
- **安装目录(C)** → 「**打开卸载程序**」(`ms-settings:appsfeatures`) / 「打开软件设置」。
- 规则 JSON 增加 `action_kind` 与 `command` 字段（向后兼容，缺省 None → 沿用文本）。

### S-E　删除能力分层（解 D5 的"快速删去"）—— 待你拍板

三个方向（详见第六节决策点）：

1. **引导式（零删除）**：只做 S-D 的运行命令/定位，CleanScope 永不删。最稳，守住现行宪法。
2. **回收站可恢复删除**：放开安全闸门 `C1`，对 **A 级 +（确认后）B 级文件夹缓存** 提供「移入回收站」（可恢复、非永久删除），严格走 `C2–C10`：系统关键/黑名单/占用/D-E 一律拒、删除前写审计、批量二次确认。
3. **二者结合（推荐）**：命令型→运行官方命令；文件夹型→移入回收站(确认+审计)；安装目录→卸载程序；全程「定位」兜底。

> 红线不破：**永远没有"永久删除"**（只进回收站可恢复）；系统关键黑名单一字不动；AI 不碰删除判定；
> 现有静态门禁 `NoPermanentDeleteTests` 仍禁 `File.Delete/Directory.Delete/SHFileOperation` 等永久删除 token，
> 回收站删除走 `Microsoft.VisualBasic.FileIO.FileSystem.Delete*(..., RecycleOption.SendToRecycleBin, ...)` 这一**可恢复**接口（需把它从禁用清单中**显式豁免**并单独测试包裹）。

### S-F　按软件聚合视图（解 D4，可选增强）

- 新增「**按软件**」视图：把归属相同的项归并，展示「<软件> —— 共 X GB，其中可清理 Y GB」，点开看该软件名下的缓存/数据/安装分布。
- 直接回答"我的空间被哪些软件占了 + 各能清多少"，比按风险更贴近用户语言。

## 四、四桶 × 现有 A–E 的映射规则（实现锚点）

| 路径特征（无系统关键 D 规则命中时） | 桶 | 内部风险 |
|---|---|---|
| 顶层容器目录（S-B 集合） | 容器 | （新增 IsContainer，不计风险） |
| 命中可清理规则 / 缓存启发（cache/temp/log…） | 可清理 | A / B |
| 命中归因（任意 owner）但无清理方式 | 谨慎 | C（含归属） |
| AppData / Program Files / ProgramData 下应用或安装目录 | 谨慎 | C |
| 个人数据（Documents/Desktop…） | 谨慎 | C |
| 系统关键黑名单 | 勿动 | D |
| 真正无规则、无归因、无缓存特征 | 勿动 | E |

> 关键变化：**E 只剩"真正三无"**（无规则/无归因/无缓存）。预计 E 从 32 项/34 GB 大幅收敛。

## 五、安全宪法影响评估（逐条确认不破红线）

| 红线 | 本轮是否触碰 | 说明 |
|---|---|---|
| ① AI 不直接删除重要文件 | 否 | AI 仍只解释/调查，删除由用户点击、且只到回收站 |
| ② 优先解释来源/依赖/风险 | 强化 | 归因前置 + AI 调查未知，正是"优先解释" |
| ③ 用户做最终删除决定 | 保持 | 任何删除都需用户显式点击 + 二次确认 |
| ④ 安全/可解释/可测试/可扩展 | 保持 | 动作结构化、回收站可恢复、审计先写后删、用例覆盖 |
| 无永久删除 API（IR-1/SR-3） | 保持 | 仅回收站可恢复删除；永久删除 token 仍被静态门禁禁止 |
| 系统关键黑名单恒 D、不可删 | 保持 | 容器/归因改动均在"无 D 规则命中"分支之后 |
| 脱敏出云（PR-1/IR-7） | 保持 | AI 增强仍走脱敏网关 |
| T-01~T-20 + 架构门禁 | 扩充 | 新增"回收站仅 A/B、D/E/系统关键/占用必拒、删除前审计"等用例 |

## 六、落地路线图与验收

| 步 | 任务 | 解决 | 价值 | 风险 | 依赖决策 |
|---|---|---|---|---|---|
| **1** | S-A 归因前置 + 有主即非 E + 字典扩充 | D3/D4 | 高（E 大降、显归属） | 低 | 无 |
| **2** | S-B 容器识别降噪（IsContainer） | D2 | 高（去噪声） | 低 | 无 |
| **3** | S-D 动作结构化 + 命令型「运行/复制」+ 文件夹「定位」 | D5/D7 | 高（可操作） | 低 | 无 |
| **4** | 四桶 UI 呈现（列表/详情/报告按桶） | D6 | 中 | 低 | 无 |
| **5** | S-C AI 真实定位（详情强化 +（可选）E 项批量调查） | D1 | 中 | 中 | 无 |
| **6** | **S-E 回收站删除**（A/B，闸门 C1–C10，确认+审计） | D5 | 高 | **中高** | **需拍板（见下）** |
| **7** | S-F 按软件聚合视图 | D4 | 中 | 中 | 无 |

**验收（关键）**：
- E 级数量较当前（32）显著下降；容器目录不再出现在风险/可清理列表。
- 任一可清理项可在**不离开应用**的情况下完成（运行命令 / 定位 / 回收站）。
- 列表/详情第一眼即见「归属软件 + 能不能删 + 怎么删」。
- 若做第 6 步：删除仅进回收站；对系统关键/黑名单/占用/D-E 发起删除**全部被拒**并审计；新增用例随 CI 门禁。

> 步骤 1–5、7 不依赖删除决策，可立即推进；第 6 步等你定方向。

## 七、需要你拍板的决策点

**删除模型三选一**（其余 E 减负、归因前置、四桶、动作闭环我都会做）：

- **A. 引导式（零删除）**：只运行官方命令 / 定位，绝不由我们删。最保守。
- **B. 回收站可恢复删除**：A/B 项加「移入回收站」，严格走安全闸门 + 确认 + 审计。
- **C. 二者结合（推荐）**：命令型跑官方命令、文件夹型进回收站、安装目录跳卸载，定位兜底。

选定后我按路线图依次实现（每步：编码 → 测试 → 提交）。
