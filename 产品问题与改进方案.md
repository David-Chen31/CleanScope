# CleanScope 产品问题与改进方案

> 基于 2026-06-16 两次真机扫描结果的诊断：
> - 全盘 `C:\`（175.14 GB / 116 万项 / Admin / 用时约 21 分钟）→ [report/CleanScope-报告-20260616-224654.md](report/CleanScope-报告-20260616-224654.md)
> - `…\AppData\Local\Temp`（211.16 MB / 1591 项 / Normal / 用时约 12 分钟）→ [report/report.md](report/report.md)
>
> 用户核心反馈：**“现在的程序对清理 C 盘没什么帮助。”** 本文定位问题、给出方案，所有方案都不破坏四条产品宪法（零删除 / 不绕过规则 / 可解释 / 用户决策）。

---

## 一、用户直接反馈的两个问题

1. **只扫小文件夹时，基本查不到风险等级和归属程序**（信息不足）。
2. **报告无法浏览/遍历**，缺少“帮助判断可删”的可视化形式（如很多软件把整盘画成矩形树图 treemap，点击下钻，矩形大小=文件大小）。

## 二、从真机报告里实测到的问题（带证据）

### P1　父目录重复计数 → 统计与 TopN 严重失真【根因级】

- 全盘报告里 `分析项占用` 之和 = C(122.66) + D(121.31) + E(583.87) = **827.84 GB**，而磁盘实际只有 **175 GB**。
- Temp 报告里 **A 级“可清理估算 523.87 MB” > 整个 Temp 文件夹 211.16 MB**。
- TopN 前 6 名是一条嵌套链：`C:\`(175G) → `C:\Users`(64G) → `C:\Users\28170`(64G) → `…\AppData`(50G) → `…\Roaming`(27G) → `…\Local`(22G)，**同一批字节在每一层被重复计入**。

> 后果：所有“占用统计 / 可清理估算 / TopN”都不可信，用户根本无法判断“空间到底去哪了”。这是当前最致命的问题。

### P2　默认 E“无法判断”淹没一切 → 对清理零指导

- 全盘：A=0、B=3（2.99 GB）、C=80、D=24、**E=93**。可清理估算仅 **2.99 GB / 175 GB ≈ 1.7%**。
- 绝大多数大目录（`Program Files`、各 `AppData` 子目录、开发缓存）都落 **E**，因为它们是“无规则命中的普通目录”，风险引擎为安全起见一律兜底 E。
- 安全方向没错，但产品**没有把真正能回收的空间挖出来**，用户看到的是一面“什么都别删”的墙。

### P3　大量典型“可清理目标”被埋没（错判为 E/C，未聚合）

报告里其实**藏着**很多业界公认的可清理项，却没被识别成 A/B：

| 真机里出现的目录 | 当前判级 | 实际性质 |
|---|---|---|
| `…\.nuget\packages`（1.49 GB） | **B ✅（唯一判对的）** | 可重建包缓存 |
| `…\.cargo\registry`、`…\.rustup`、`…\.gradle` | E | 可重建工具链缓存 |
| `…\Miniconda3\pkgs` | E | conda 包缓存（可 `conda clean`） |
| `…\AppData\Local\Microsoft\vscode-cpptools\ipch` | E | C++ 预编译头缓存（可重建） |
| `…\Roaming\Code\WebStorage`、`workspaceStorage` | C | 编辑器缓存/工作区缓存 |
| 各类浏览器/Electron `Cache / GPUCache / Code Cache` | C/E | 可重建缓存 |

> 即“规则覆盖不足 + 缺少目录级缓存识别”，导致最有价值的省空间机会沉没。

### P4　归属（Attribution）对用户数据目录普遍失效 → 印证用户问题①

- `…\Roaming\Tencent`、`LarkShell`、`s3_web`、`QQEX`、`Notion` 等，**归属一栏是“-”**。
- 根因：归因只靠“已安装应用注册表 InstallLocation 前缀匹配”，而 `AppData\Roaming\*` 不是安装目录，匹配不上。
- 只扫一个小文件夹时，还**丢失了父级上下文**（它在 `AppData\Roaming` 下、对应哪个已装程序、目录内有无带签名的 exe），于是风险默认 C/E、归属空白。

### P5　报告形式不可浏览 → 印证用户问题②

- 95 KB 扁平 Markdown、几百行平铺，**无层级、无大小条、无 treemap、无下钻、无筛选**。
- WPF 列表虽能按大小/风险排序，但仍是一维长列表，无法回答“空间在树里怎么分布”。

### P6　性能：逐项串行云端 AI 调用 → 百量级项就要十几分钟【体验级】

- Temp 仅 211 MB / 1591 项，却耗时 **约 12 分钟**；全盘约 21 分钟。
- 瓶颈基本可定位为：**每个分析项串行发起一次云端 AI 解释**（约 100–200 项 × 每次数秒 ≈ 10–20 分钟）。
- 对比 WizTree 直接读 NTFS MFT，全盘可在**数秒**完成扫描。当前“先把所有 TopN 都 AI 解释一遍”的策略，既慢又贵。

### P7　缺少“按类别聚合”的清理视图

- CCleaner / BleachBit 的核心是**按类别聚合**：“浏览器缓存 X GB”“临时文件 Y GB”“回收站 Z GB”，预览后一键处理。
- CleanScope 把每个文件单独列一行，**没有“这一类总共能省多少”**，用户得自己在几百行里大海捞针。

## 三、根因总结

当前产品把自己定位成 **“对每个文件逐项判风险”** 的分类器，并把未知一律兜底为 E。这导致：

- 它**不是**一个好的**空间可视化工具**（没有 treemap/下钻，父目录还重复计数）；
- 也**不是**一个好的**垃圾清理工具**（没把已知可回收类别聚合成“可省 X GB”）；
- 它做的“第三件事”——给整盘每个节点贴风险标签——产出的恰恰是用户最不需要的“满屏别删”。

用户其实想要两种心智模型，而我们一个都没满足好：
**“我的空间去哪了？”（空间地图）** 与 **“我能安全省出多少？”（清理建议）**。

## 四、业界是怎么做的（联网调研）

- **空间可视化类（WizTree / WinDirStat / TreeSize / SpaceSniffer）**：核心是 **treemap 矩形树图 + 可下钻目录树 + 最大文件/文件夹排行 + 文件类型分布**，让人“一眼看出大头在哪”。WizTree 直接读 **NTFS MFT**，全盘扫描数秒完成（WinDirStat/TreeSize 走文件系统 API，慢但通用）。它们**只回答“空间去哪了”，不替你判断能不能删**。
- **垃圾清理类（CCleaner / BleachBit）**：维护一份**已知安全可清理位置**清单（浏览器缓存、临时目录、缩略图缓存、日志、崩溃报告、Windows 更新残留、回收站…），**保守地只清理已知位置、不猜测每个程序的垃圾在哪**，并且**删除前一律先预览**。

> 关键启发：**“可视化定位空间”** 与 **“按已知类别给可清理项”** 是两套独立能力；安全的做法是“只对已知类别下结论 + 删除前预览”，而不是“对全盘每个文件强行判一个风险等级”。

## 五、解决方案（尊重四宪法，分层推进）

### 总纲：把单一“风险表”重构为三种视图

| 视图 | 回答的问题 | 对标 |
|---|---|---|
| **A. 空间地图** | 我的空间去哪了？ | WizTree/WinDirStat treemap + 目录树下钻 |
| **B. 清理建议** | 我能安全省出多少？ | CCleaner/BleachBit 按类别聚合 + 预览 |
| **C. 详情/审计** | 这一项到底是什么、凭什么这么判？ | 现有逐项分析（保留，但不再当主视图） |

安全前提不变：A/B 也**绝不自动删**，只给“官方清理命令/步骤”，删除决策永远在用户。

---

### S1【最高优先·根因】修复父目录重复计数

- 统计/TopN 改用 **独占大小（exclusive / leaf 贡献）**，或明确以**树**呈现层级，**不要把祖先目录与其子目录并列进同一张 TopN/统计**。
- “风险统计的占用”“可清理估算”必须**按互斥子树求和**（同一字节只计一次）。
- 验收：Temp 报告里 A 级估算不再超过文件夹总大小；全盘各级占用之和 ≤ 磁盘实际占用。

### S2【高优先·解用户问题②】引入 treemap + 目录树下钻（WPF）

- 扫描产出**完整目录树（或可下钻的分层聚合）**，WPF 增加一个 **treemap 视图**：矩形面积=大小、颜色=风险或文件类型，点击下钻、面包屑回溯。
- 可用 WPF `Canvas` 自绘 squarified treemap，无需第三方依赖（与现有“无第三方框架”取向一致）。
- 同时提供经典“目录树 + 大小条”侧栏，作为 treemap 的列表镜像。

### S3【高优先·解“没帮助”】建“可清理类别”知识库 + 聚合视图（视图 B）

- 扩充 `rules/`，把以下标为**可清理类别（A/B）**并带官方清理方式：
  - 开发缓存：`.nuget`、`.cargo`、`.gradle`、`.rustup`、`npm-cache`、`pip cache`、`conda pkgs`、`yarn`、`go\pkg\mod`、`vscode-cpptools\ipch`、JetBrains/VS 缓存；
  - 应用缓存：浏览器/Electron `Cache/GPUCache/Code Cache/blob_storage`、缩略图缓存；
  - 系统垃圾：`Windows\Temp`、用户 `Temp`、`SoftwareDistribution\Download`（更新残留）、回收站、崩溃转储、日志。
- 新增**“按类别聚合”视图**：每类显示**总可回收大小 + 风险 + 官方/安全清理方式 + 可展开看明细**（对齐 CCleaner/BleachBit，但只给命令不直删）。
- 这是把“2.99 GB 可清理”变成“真实可回收几十 GB 机会”的关键一步。

### S4【高优先·解用户问题①】增强归因与小文件夹上下文

- **多线索归因**（在现有注册表匹配之外补充）：
  - **路径模式归因**：`AppData\{Roaming,Local}\<X>`、`AppData\Local\Packages\<PkgFamily>`、`.cargo/.rustup/.nuget/.gradle` → 直接得出应用/工具链标识；
  - **可执行/签名线索**：目录内 `.exe` 的 Authenticode 签名者/产品名；同名运行进程；
  - **常见厂商目录字典**：Tencent/Lark/Notion/JetBrains… 目录名 → 应用名（纳入知识库）。
- **小文件夹扫描自动补父级上下文**：向上回溯到 `AppData / Program Files / ProgramData` 边界，用父级路径模式参与归因与风险判定，避免“信息不足→全 C/E/空归属”。

### S5【中优先】风险引擎：识别“缓存/可重建”，给 E 减负

- 对“无规则命中的目录”不再一律 E。增加启发式：目录名含 `cache/caches/temp/tmp/logs/Crashpad/GPUCache/Code Cache/blob_storage/ipch` 等 → 倾向 **A/B（可重建）**；包/工具链目录 → **B**。
- **E 只留给真正取证失败/异常**，而非“普通未知目录”。保持“宁保守”，但把“明显可重建的缓存”从 E 里救出来。

### S6【中优先·解性能】AI 改为按需 + 缓存 + 并发；扫描与取证解耦

- **扫描阶段只算大小树**（快），**不逐项 AI**；AI 解释**改为打开详情时按需触发**，或仅对“用户聚焦的少数项”调用。
- 已解释结果**按脱敏路径模式缓存**（同类目录复用，避免重复调用）。
- 需要批量解释时**并发 + 限流**，而非串行。
- 进阶（可选）：全盘“空间地图”场景直接**读 NTFS MFT** 加速到秒级，取证/判风险**下钻时惰性进行**。

### S7【低优先】报告形式增强

- Markdown：新增“按类别聚合”段；用**缩进树**表达层级（替代扁平平铺）；TopN 改“**叶子贡献** Top”，不再并列祖先。
- 新增**可交互 HTML 报告**（内嵌 treemap）与 **CSV 导出**（对齐 WizTree，便于 Excel 透视）。

## 七、与安全宪法的关系（重要）

以上全部方案都**不触碰红线**：

- 仍然**零删除**：视图 B 的“清理建议”只给官方命令/步骤与“加入忽略名单”，删除入口依旧不存在。
- AI 仍是**旁路**：把缓存识别交给**规则引擎**（声明式知识库），不是让 AI 判“能不能删”。
- E 减负**不等于放松**：只把“可重建缓存”从 E 移到 A/B，系统关键黑名单（D）一字不动。
- 所有改动仍由 **T-01 ~ T-20 安全测试 + CI 硬门禁**守护。
