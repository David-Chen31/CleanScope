> 上游依据：[需求冻结文档.md](需求冻结文档.md)、[架构设计.md](架构设计.md)（六模块裁决链 + 四外围解耦：前端/AI/规则库/存储）。

## 0. 一句话结论

> **推荐：.NET 8 + C# + WPF（MVVM）+ Microsoft.Data.Sqlite。**
> 理由：CleanScope 本质是**重度依赖 Windows 系统集成的桌面系统工具**——注册表、进程占用、数字签名、Appx/WinGet、文件元数据这些"取证"能力，.NET 是原生一等公民，而其他栈都要绕路。选成熟栈，就该选与目标平台贴合最紧的那个。

## 2. 推荐技术栈（详细）

### 2.1 主栈

| 层 | 技术 | 说明 |
|---|---|---|
| 运行时 | **.NET 8（LTS）** | 长期支持，性能强 |
| 语言 | **C# 12** | 强类型，团队/AI 协作可读性好 |
| UI | **WPF + MVVM** | 成熟稳定；用 `CommunityToolkit.Mvvm` 做 MVVM |
| DI/宿主 | `Microsoft.Extensions.DependencyInjection` + `Hosting` | 支撑分层解耦与接口隔离 |
| 存储 | **Microsoft.Data.Sqlite**（经 `IStorage` 接口） | 轻量、可控；不强绑 EF |
| 系统访问 | `Microsoft.Win32.Registry`、`System.Diagnostics.Process`、`System.Security.Cryptography.X509Certificates`（Authenticode）、`System.Management`（WMI）、`Windows.Management.Deployment`（Appx）、WinGet COM/CLI | 取证能力原生 |
| AI | `HttpClient` 调云端 LLM；本地模型经统一 `IExplanationService` 接口 | 满足 AI 可降级硬要求 |
| 日志 | `Microsoft.Extensions.Logging` + Serilog | 审计与诊断 |
| 测试 | xUnit + FluentAssertions + Moq | 安全红线测试门禁 |
| 打包 | MSIX 或单文件自包含 exe | 发布 |

### 2.2 UI 子选择：WPF vs WinUI 3

| | WPF | WinUI 3 |
|---|---|---|
| 成熟度/稳定性 | ◎ 十余年沉淀，问题可查 | ○ 现代但生态仍在打磨 |
| 外观现代感 | ○（可用 WPF-UI 等库美化） | ◎ Fluent 原生 |
| 适配本项目 | **推荐起步**：稳、文档多、坑少 | 可选：追求现代观感时 |

> **裁决：MVP–Beta 用 WPF 求稳；若后期要现代 Fluent 外观，因 UI 已与核心解耦（架构决议①），可平滑替换 UI 层而不动裁决链。** 这正是架构"前端可替换"的价值兑现点。

## 3. 不推荐方案及原因

| 方案 | 不推荐原因 |
|---|---|
| **Tauri + Rust** | 取证能力要大量 `winapi`/`windows-rs` FFI，开发量与门槛陡增；Rust 学习曲线对当前团队偏高，拖慢成熟产品交付。体积小的优势不足以抵消系统集成成本。 |
| **Electron** | 系统集成最弱（注册表/签名/进程需原生模块），Node 攻击面大与本项目"安全优先"相悖，包体臃肿。不适合系统工具。 |
| **Python + PySide6** | 快速原型很好，但定位是"成熟产品栈"则不合适：动态类型维护成本高、打包体积大、深度 Windows 集成需大量 pywin32 胶水、产品化天花板低。**与本次"一步到位"取向冲突，故不选。** |

## 4. 第一版目录结构

按架构分层组织，**让目录结构直接体现"六模块裁决链 + 四外围解耦 + 单一安全闸门"**。采用 .NET 多项目方案（Solution）。

```text
CleanScope/                          # 仓库根 (git init 于此)
├─ CleanScope.sln                    # 解决方案
├─ docs/                             # 现有设计文档 (计划书/冻结/架构/本文件…)
├─ rules/                            # 声明式规则库 (数据, 非代码)
│  ├─ 00-system-critical.json
│  ├─ 10-windows-installer.json
│  └─ ... (见架构 §7)
├─ src/
│  ├─ CleanScope.Domain/             # 【核心】领域模型 + 六模块接口, 零外部依赖
│  │  ├─ Entities/                   #   ScanTask/FileNode/Evidence/RiskAssessment...
│  │  ├─ Abstractions/               #   IStorage, IExplanationService, IRuleSource...
│  │  └─ Safety/                     #   安全契约 (风险等级/红线常量)
│  ├─ CleanScope.Core/               # 【核心】六模块实现 (裁决链)
│  │  ├─ Scan/                       #   ScanEngine
│  │  ├─ Evidence/                   #   EvidenceCollector
│  │  ├─ Rules/                      #   RuleEngine (加载 rules/, 权威)
│  │  ├─ Attribution/                #   AttributionEngine
│  │  ├─ Risk/                       #   RiskEngine (权威)
│  │  └─ Decision/                   #   DecisionService
│  ├─ CleanScope.Safety/             # 【安全闸门】SafetyGuard + ActionExecutor (唯一可改盘路径)
│  ├─ CleanScope.Ai/                 # 【AI 旁路】SanitizationGateway + ExplanationService + Validator
│  ├─ CleanScope.Infrastructure/     # 【外围】WindowsAccess(注册表/进程/签名/Appx) + SQLite 实现 IStorage
│  ├─ CleanScope.Reporting/          # 【外围】报告导出 (md/html/json/csv)
│  └─ CleanScope.App.Wpf/            # 【前端】WPF + MVVM, 仅依赖 Domain 接口, 经 DI 组装
├─ tests/
│  ├─ CleanScope.Core.Tests/         # 六模块单测
│  ├─ CleanScope.Safety.Tests/       # 🔴 安全红线测试 (SR-1~10 门禁)
│  └─ CleanScope.Ai.Tests/           # AI 校验器/脱敏测试
├─ .gitignore                        # 忽略 bin/obj/本地 db/密钥/扫描结果
└─ README.md
```

**依赖方向（强制单向，防架构腐化）：**
```text
App.Wpf ─▶ Domain ◀─ Core ─▶ Domain
                  ◀─ Safety, Ai, Infrastructure, Reporting
```
- **Domain 不依赖任何其他项目**（最内核，含接口与领域模型）。
- Core/Safety/Ai/Infrastructure 依赖 Domain 接口，互不直接依赖。
- App.Wpf 只依赖 Domain 抽象，运行时经 DI 注入具体实现 → **换 UI / 换存储 / 换 AI 实现都不碰核心**。
- **CleanScope.Ai 在编译期就不引用 CleanScope.Safety** → 从项目引用层面坐实"AI 无法触发删除"(架构 A-2)。

> 这个结构让"AI 旁路""单一安全闸门""存储接口隔离"成为**编译期可验证**的约束，而非口头纪律。

## 5. 技术迁移规划

**取向已定为"一步到位"，故无主动迁移计划。** 但架构保留两处低成本演进路径：

| 演进 | 触发条件 | 成本 |
|---|---|---|
| WPF → WinUI 3 | 需现代 Fluent 外观 | 仅替换 `App.Wpf`，核心零改动 |
| 核心服务进程化 | 未来需前后端分离/远程 | Domain 接口已就绪，加 IPC 层即可 |
| SQLite → 其他存储 | 企业版/大数据量 | 仅替换 Infrastructure 的 IStorage 实现 |

### 7.1 决议落地校验点（如何保证不被违反）
- **决议 3｜.NET 10 预留**：在 `Directory.Build.props` 集中定义 `<TargetFramework>`，全解决方案统一引用，升级时改一处。
- **决议 9｜核心不依赖 WPF**：`CleanScope.Domain` / `CleanScope.Core` 的项目文件**不引用** `CleanScope.App.Wpf` 或任何 WPF 程序集；可加架构测试（如 NetArchTest）在 CI 中断言。
- **决议 10｜AI 不入裁决链**：`CleanScope.Ai` 项目**不引用** `CleanScope.Safety`；AI 输出只经 Validator 流向 DecisionService，结构上无法触发删除。

> 技术基线已定稿，进入 **阶段④ 安全模型设计**（`安全设计.md` + `风险分级细则.md`）。
> 开发环境：.NET 8 SDK + Visual Studio 2022（或 VS Code + C# Dev Kit）。
```
